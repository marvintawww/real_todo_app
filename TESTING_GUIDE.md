# 📘 Полное руководство по тестированию для FastAPI проекта

---

## 1. Что такое тесты и зачем они нужны

### Определение

**Тесты** — это отдельный код, который проверяет, работает ли твой основной код правильно. Вместо того чтобы вручную запускать приложение и проверять каждую функцию, ты пишешь тесты, которые делают это автоматически.

### Зачем нужны тесты

| Без тестов | С тестами |
|------------|-----------|
| Баги находят пользователи | Баги находят тесты до продакшена |
| Страшно рефакторить | Уверенно меняешь код |
| Ручная проверка занимает время | Автоматическая проверка за секунды |
| Зависимость от конкретного разработчика | Любой видит что сломалось |

## 2. Виды тестов (Пирамида тестирования)

```
        /\
       /  \      E2E (End-to-End)
      /____\     Мало, дорого, медленно
     /      \
    /        \   Integration (Интеграционные)
   /__________\  Среднее количество
  /            \
 /              \ Unit (Много, дешево, быстро)
/________________\
```

### 2.1 Unit-тесты (Модульные)

**Что тестируют:** Отдельные функции, методы или классы в изоляции.

**Пример для твоего проекта:**
```python
# ТЕСТИРУЕМ: UserService.create_user()
# Вопрос: "Создается ли пользователь с корректными данными?"
```

**Характеристики:**
- Быстрые (миллисекунды)
- Не зависят от БД, сети, файловой системы
- Тестируют одну "единицу" логики
- Много их должно быть (70-80% от всех тестов)

**В твоём проекте тестируем:**
- `UserService` — создание, получение, обновление пользователя
- `TaskService` — создание, завершение, удаление задач
- `JWTService` — генерация и валидация токенов

### 2.2 Integration-тесты (Интеграционные)

**Что тестируют:** Взаимодействие между несколькими компонентами.

**Пример:**
```python
# ТЕСТИРУЕМ: Как UserRepository работает с PostgreSQL
# Вопрос: "Сохраняется ли пользователь в БД корректно?"
```

**Характеристики:**
- Медленнее чем unit (нужна БД)
- Проверяют "швы" между компонентами
- Среднее количество (15-20%)

**В твоём проекте тестируем:**
- Репозитории с реальной/тестовой БД
- API эндпоинты через TestClient

### 2.3 E2E-тесты (End-to-End)

**Что тестируют:** Приложение целиком, как пользователь.

**Пример:**
```python
# ТЕСТИРУЕМ: Полный сценарий
# 1. Регистрация -> 2. Логин -> 3. Создание задачи -> 4. Проверка
```

**Характеристики:**
- Медленные
- Дорогие в поддержке
- Малое количество (5-10%)

## 3. Порядок действий при разработке с тестами (TDD)

### Что такое TDD (Test-Driven Development)

TDD — это подход, где тесты пишутся ДО кода.

### Цикл TDD (Red-Green-Refactor)

```
1. RED    → Пишешь тест, который падает (потому что кода нет)
2. GREEN  → Пишешь минимум кода, чтобы тест прошел
3. REFACTOR → Улучшаешь код, не ломая тесты
4. Повторяешь
```

### Пример для твоего проекта

Допустим, нужно добавить метод `delete_user` в `UserService`:

**Шаг 1: RED (Пишешь тест)**
```python
# tests/unit/test_services/test_user.py
async def test_delete_user_success():
    # Тест описывает, как ДОЛЖНА работать функция
    # Тест упадет, потому что метода delete_user еще нет
    pass
```

**Шаг 2: GREEN (Пишешь код)**
```python
# src/services/user.py
async def delete_user(self, user_id: int):
    # Минимум кода чтобы тест прошел
    # Можно даже просто: pass
    pass
```

**Шаг 3: REFACTOR**
```python
# Улучшаешь реализацию
async def delete_user(self, user_id: int) -> User:
    user = await self.get_user(user_id)
    user.is_active = False
    return user
```

### Почему TDD полезен

- Сразу думаешь о том, что хочешь получить
- Код сразу тестируемый
- Меньше "костылей"
- Тесты пишутся, а не откладываются "на потом"

## 5. Что такое Mock (мок) и зачем он нужен

### Проблема

Unit-тест должен работать в изоляции. Но твой сервис зависит от:
- Репозитория (который ходит в БД)
- Других сервисов

### Решение: Mock

**Mock** — это "подделка" зависимости, которая возвращает нужные данные без реального выполнения.

### Пример

```python
# БЕЗ MOCK (интеграционный тест, медленный)
async def test_create_user():
    db = RealDatabase()  # Реальная БД
    repo = UserRepository(db)
    service = UserService(repo)
    user = await service.create_user(...)  # Идет в PostgreSQL

# С MOCK (unit тест, быстрый)
async def test_create_user():
    mock_query = MagicMock()  # Подделка
    mock_query.get_by_login.return_value = None  # Всегда возвращает None
    mock_command = MagicMock()
    
    service = UserService(mock_query, mock_command)
    user = await service.create_user(...)  # Не ходит в БД!
```

### Когда использовать моки

| Зависимость | Мок? |
|-------------|------|
| База данных | ✅ Да (в unit тестах) |
| Внешний API | ✅ Да |
| Другие сервисы | ✅ Да |
| Внутренняя логика (другой сервис в том же проекте) | ❌ Лучше интеграционный тест |

## 7. Что такое Fixture (фикстура)

### Определение

**Fixture** — это подготовленные данные или объекты для тестов.

### Пример

```python
# conftest.py
import pytest
from unittest.mock import MagicMock

@pytest.fixture
def mock_user_repository():
    """Фикстура создает мок репозитория"""
    mock = MagicMock()
    mock.get_by_id.return_value = None
    return mock

# В тесте
async def test_something(mock_user_repository):
    # mock_user_repository уже создан и доступен
    pass
```

### Типы фикстур

| Тип | Пример | Когда использовать |
|-----|--------|-------------------|
| `session` | pytest | Для каждого теста создается новая сессия |
| `module` | pytest | Одна на весь файл |
| `function` | pytest | По умолчанию, для каждого теста |

## 8. Практический план для твоего проекта

### Этап 1: Подготовка (1-2 дня)

1. **Создай файл `tests/conftest.py`**
   - Добавь фикстуры для моков `query` и `command`
   - Настрой pytest-asyncio

2. **Создай файл `pytest.ini`**
   ```ini
   [pytest]
   asyncio_mode = auto
   python_files = test_*.py
   ```

### Этап 2: Первые Unit-тесты (3-5 дней)

Начни с `UserService`:

1. **Тест: `create_user` успешный**
   - Мокни `query.get_by_login` → возвращает `None`
   - Мокни `command.create` → возвращает созданного пользователя
   - Проверь: метод вызывает нужные зависимости

2. **Тест: `create_user` когда пользователь уже существует**
   - Мокни `query.get_by_login` → возвращает пользователя
   - Проверь: выбрасывается `ItemAlreadyExist`

3. **Тест: `get_user` успешный**
   - Мокни `query.get_by_id` → возвращает активного пользователя

4. **Тест: `get_user` когда пользователь не найден**
   - Мокни `query.get_by_id` → возвращает `None`
   - Проверь: выбрасывается `ItemNotExist`

### Этап 3: Интеграционные тесты репозиториев (3-5 дней)

1. **Настрой тестовую БД** (SQLite в памяти для скорости)
2. **Тестируй `UserRepository.get_by_login`**
3. **Тестируй `UserRepository.create`**

### Этап 4: Интеграционные тесты API (2-3 дня)

1. **Тестируй эндпоинты через `TestClient`**
2. **Проверь регистрацию, логин, создание задачи**

## 9. Ключевые принципы хороших тестов

### AAA Pattern (Arrange-Act-Assert)

```python
async def test_example():
    # Arrange (Подготовка)
    mock_repo = MagicMock()
    service = UserService(mock_repo)
    
    # Act (Действие)
    result = await service.get_user(1)
    
    # Assert (Проверка)
    assert result is not None
```

### Другие принципы

1. **Один тест — одна проверка** — лучше много маленьких тестов, чем один большой
2. **Имя теста говорит что тестируется** — `test_create_user_raises_error_when_login_exists`
3. **Тесты не должны зависеть друг от друга** — каждый работает сам по себе
4. **Тесты должны быть быстрыми** — если медленные, будут лениться запускать

## 10. Как запускать тесты

```bash
# Запустить все тесты
pytest

# Запустить конкретный файл
pytest tests/unit/test_services/test_user_service.py

# Запустить с покрытием
pytest --cov=src --cov-report=html

# Запустить только тесты определенного маркера
pytest -m unit      # Только unit тесты
pytest -m integration  # Только интеграционные
```

## 12. Частые ошибки начинающих

| Ошибка | Почему плохо | Решение |
|--------|--------------|---------|
| Тесты зависят друг от друга | Падает случайно | Каждый тест сам готовит данные |
| Тестируют внутренности | Ломается при рефакторинге | Тестируй только публичные методы |
| Много assertion в одном тесте | Сложно понять что сломалось | Один assert — один тест |
| Не запускают тесты | Тесты становятся бесполезными | Запускай после каждого изменения |
